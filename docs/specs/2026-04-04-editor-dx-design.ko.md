# orv 내장 에디터 및 DX 설계

> 날짜: 2026-04-04
> 상태: 초안
> 대상: 언어, 컴파일러, 에디터 구현 계획

---

## 1. 목적

이 문서는 orv가 제공하고자 하는 에디터 경험을 정의한다.

목표는 언어 완성 후에 범용 외부 IDE에 언어 도구를 덧붙이는 것이 아니다. 목표는 내장 에디터가 처음부터 최상의 편집 경험을 제공할 수 있도록 언어, 파서, 분석기, 런타임 메타데이터를 설계하는 것이다.

---

## 2. 핵심 입장

orv는 슬래시 커맨드 중심이 **아니다**.

언어가 `@node`과 `%property`를 중심으로 구축되었으므로, 에디터도 동일한 트리거를 중심으로 구축되어야 한다:

- `@`는 현재 도메인에 맞는 노드 팔레트를 연다
- `%`는 현재 노드에 맞는 프로퍼티 팔레트를 연다
- `Shift + Up` / `Shift + Down`은 단순 텍스트 이동이 아니라 시맨틱 값을 스텝한다
- 현재 커서 위치가 무엇이 유효하고, 보이고, 제안되는지를 결정한다

참조 인터랙션 목표는 layouts.dev 같은 제품의 품질이며, 슬래시 커맨드 에디터를 복사하는 것이 아니라 orv의 문법에 맞게 적용하는 것이다.

---

## 3. 에디터 원칙

### 3.1 내장 에디터 우선

최상의 orv 편집 경험은 서드파티 IDE 확장의 제약에 의존하지 않고, orv 네이티브 에디터에서 제공되어야 한다.

외부 IDE 통합은 나중에 존재할 수 있지만, 주요 제품 형태는 아니다.

### 3.2 도메인 인식 제안

에디터는 도메인 컨텍스트를 이해해야 한다:

- `@html` 내부에서는 UI 노드와 UI 안전 프로퍼티를 표시
- `@server` 내부에서는 `@listen`, `@route`, `@respond`, `@serve` 같은 서버 노드를 표시
- `@design` 내부에서는 디자인 토큰과 토큰 프로퍼티를 표시
- `@route` 내부에서는 라우트별 요청/응답 작성 헬퍼를 표시

제안은 범용 문자열 목록이 아니어야 한다. 시맨틱 컨텍스트에 의해 필터링된 유효한 선택지여야 한다.

### 3.3 커서 로컬 편집

커서 위치가 UI를 주도해야 한다:

- `@`에서는 현재 컨텍스트에 유효한 노드를 표시
- `%`에서는 현재 노드에 유효한 프로퍼티를 표시
- 프로퍼티 값에서는 해당 프로퍼티의 타입이나 토큰 공간에 기반한 값 제안을 표시
- 라우트 메서드/경로 슬롯에서는 범용 자동완성 대신 라우트 인식 어포던스를 표시

### 3.4 텍스트 트릭보다 구조적 편집

에디터는 원시 텍스트뿐만 아니라 구문 및 시맨틱 구조에 기반하여 동작해야 한다.

예시:

- `text-2xl`을 `text-3xl`로 증가
- 간격 스케일 감소
- enum 유사 프로퍼티 값 순환
- 숫자 리터럴 증가 또는 감소
- 동일 토큰 패밀리 내에서 색상 또는 크기 토큰 스텝

### 3.5 tree-sitter 수준의 정밀도와 속도

언어 서비스 목표는 tree-sitter 수준의 응답성과 구조적 정밀도이다:

- 증분 친화적 파싱
- 안정적인 바이트 스팬
- 저비용 커서 컨텍스트 쿼리
- 작은 편집에 대한 최소 재분석
- 전체 시맨틱 분석 완료 전에도 에디터 작업에 충분한 구문 트리

이것은 tree-sitter 자체를 임베딩해야 한다는 의미가 **아니다**. 정밀도, 지연 시간, 견고성에서 동등한 품질을 요구한다는 의미이다.

## 4. 필수 사용자 인터랙션

### 4.1 `@` 노드 팔레트

`@`를 입력하면 도메인별로 필터링된 항목이 있는 팔레트가 열려야 한다.

예시:

- `@html` 내: `@div`, `@text`, `@button`, `@input`, `@body`
- `@server` 내: `@listen`, `@route`, `@before`, `@after`
- `@route` 내: `@respond`, `@serve`

각 항목에는 다음이 포함되어야 한다:

- 짧은 설명
- 허용되는 컨텍스트
- 스니펫 형태
- 주요 프로퍼티

### 4.2 `%` 프로퍼티 팔레트

`%`를 입력하면 현재 노드에 유효한 프로퍼티를 표시해야 한다.

예시:

- `@button`: `%onClick`, `%class`, `%style`
- `@input`: `%value`, `%placeholder`, `%type`, `%onInput`
- `@respond`: `%header`

에디터는 다음을 구분해야 한다:

- 인라인 `%` 프로퍼티
- 들여쓰기 연속 `%` 프로퍼티
- 노드 본문 내부의 `%` 프로퍼티

### 4.3 시맨틱 스테퍼

커서가 스텝 가능한 토큰 위에 있을 때, `Shift + Up` / `Shift + Down`은 토큰을 시맨틱하게 변환해야 한다.

예시:

- `text-sm -> text-base -> text-lg -> text-xl`
- `p-2 -> p-3 -> p-4`
- `200 -> 201 -> 202`
- `true <-> false`

스테퍼는 프로퍼티를 인식해야 한다. 크기 스테퍼가 관련 없는 식별자에서 활성화되어서는 안 된다.

### 4.4 라우트 인식 편집

`@route` 내부에서 에디터는 다음을 이해해야 한다:

- 메서드 슬롯
- 경로 슬롯
- 라우트 본문
- 유효한 respond/serve 노드

가능한 UI 동작:

- 메서드 드롭다운
- 경로 파라미터 시각화
- respond 헬퍼 스니펫

### 4.5 힌트 인식 편집

`@hint`는 대상 컨텍스트에 따라 유효한 키와 값을 노출해야 한다.

예시:

- `@hint protocol=` -> `json`, `binary`, `hybrid`
- `@hint render=` -> `ssr`, `csr`, `ssg`
- `@hint chunk=` -> `separate`, `inline`

## 5. 컴파일러 및 언어 서비스 요구사항

위의 에디터를 구현하려면, 컴파일러 스택은 최종 HIR 덤프 이상을 노출해야 한다.

### 5.1 구문 요구사항

구문 레이어는 다음을 제공해야 한다:

- 모든 관련 노드와 토큰에 대한 안정적인 바이트 스팬
- 오류 허용 파스 결과
- 부분적으로 유효하지 않은 코드에서도 노드/프로퍼티 경계
- 작은 편집에 대한 증분 재파싱 전략
- 전체 컴파일 없이 커서 쿼리에 응답할 수 있는 충분한 구조

### 5.2 시맨틱 요구사항

분석기는 다음을 제공해야 한다:

- 커서 위치에서의 현재 도메인 컨텍스트
- 현재 포함 노드
- 해당 컨텍스트에 유효한 자식 노드
- 해당 노드에 유효한 프로퍼티
- 프로퍼티 값 카테고리
- 코드가 불완전한 상태에서도 사용 가능한 진단

### 5.3 레지스트리 요구사항

orv는 다음에 대한 머신 리더블 메타데이터 레지스트리가 필요하다:

- 노드 정의
- 유효한 도메인
- 유효한 자식 관계
- 유효한 프로퍼티
- 프로퍼티 값 종류
- 스테퍼 동작
- 문서화 문자열

이 메타데이터는 컴파일러가 소유해야 하며, 에디터에서 별도로 복제해서는 안 된다.

## 6. 제안 아키텍처

### 6.1 새로운 에디터 대응 레이어

핵심 구문/분석기 기반이 안정화된 후 에디터 지향 크레이트를 추가한다:

| 크레이트 | 목적 |
|------|---------|
| `orv-ide` | 커서 컨텍스트, 자동완성, 호버, 스테퍼, 에디터 메타데이터 API |

### 6.2 책임 분리

- `orv-syntax`: 토큰, 스팬, 파스 트리, 증분 친화적 구문 데이터
- `orv-analyzer`: 도메인/타입 적법성, 시맨틱 컨텍스트, 심볼 정보
- `orv-ide`: 에디터 쿼리 및 UX 지향 API
- 내장 에디터 앱: 렌더링, 인터랙션, 키바인딩, 커맨드 서피스

### 6.3 최초 `orv-ide` API

초기 API 세트에는 다음이 포함되어야 한다:

- `cursor_context(file, offset) -> Context`
- `complete_at(file, offset, trigger) -> Suggestions`
- `step_value(file, offset, direction) -> Edit`
- `hover_at(file, offset) -> Info`
- `outline(file) -> Tree`

`trigger`는 최소한 다음을 지원해야 한다:

- `At`
- `Percent`
- `Manual`

## 7. 로드맵 영향

이것은 구현 우선순위를 두 가지 방식으로 변경한다:

1. 파서/분석기 작업은 에디터 수준의 구조적 정보를 보존해야 한다
2. 내장 에디터는 사후 고려가 아닌 계획된 제품 표면이다

이것은 다음을 의미한다:

- 커서 컨텍스트 영향을 무시하는 언어 변경이 없어야 한다
- 부분 코드 편집을 취약하게 만드는 파서 지름길이 없어야 한다
- 도메인/프로퍼티 메타데이터는 일급 컴파일러 출력으로 취급되어야 한다

---

## 8. 첫 번째 에디터 패스의 비목표

첫 번째 에디터 패스에는 다음이 필요하지 **않다**:

- 완전한 범용 LSP 동등성
- 가능한 모든 리팩토링
- 협업 편집
- AI 네이티브 코드 생성 UX
- 비주얼 캔버스 편집

첫 번째 패스에 **필요한** 것:

- 빠른 `@` 노드 자동완성
- 빠른 `%` 프로퍼티 자동완성
- 신뢰할 수 있는 커서 컨텍스트 감지
- 시맨틱 스테퍼
- 불완전한 코드에서의 안정적인 진단

---

## 9. 즉시 다음 단계

1. 구문 및 분석기 변경을 증분 친화적으로 유지한다.
2. 머신 리더블 노드/프로퍼티/도메인 메타데이터를 정의한다.
3. 소스 스팬에 대한 커서 컨텍스트 쿼리를 추가한다.
4. 실제 에디터 쉘을 구축하기 전에 `@` 및 `%` 자동완성 API를 추가한다.
5. 에디터 UI 코드에서 로직을 복제하지 않고 컴파일러의 메타데이터를 기반으로 내장 에디터를 설계한다.
