## 1. 목적

이 문서는 현재 orv 언어 문서와 최적화 명세를 실행 가능한 구현 로드맵으로 전환한다.

다음 다섯 가지 실질적인 질문에 답하기 위한 것이다:

1. 무엇을 먼저 구축하는가?
2. 첫 번째 사용 가능한 MVP에 명시적으로 포함되는 것은 무엇인가?
3. 각 단계 이후 어떤 크레이트와 파일이 존재해야 하는가?
4. 각 마일스톤에서 사용자에게 무엇이 동작해야 하는가?
5. 다음 단계로 넘어가기 전에 어떤 엣지 케이스와 실패 모드를 처리해야 하는가?

이 로드맵은 이제 컴파일러 내부가 향후 orv 네이티브 내장 에디터에서 직접 사용할 수 있도록 구축되어야 한다는 가정도 포함한다.

## 3. 전달 철학

### 3.1 작업 순서

- 최적화보다 정확성 우선
- 코드 생성보다 진단 우선
- 최적화 백엔드보다 참조 실행 우선
- 전송/런타임 특화보다 안정적 IR 우선
- 어댑티브 동작보다 보수적 시맨틱 우선
- 에디터 쉘 작업보다 에디터 수준 구문 데이터 우선

### 3.2 로드맵 규칙

모든 기능은 동일한 단계를 거쳐 졸업한다:

1. 구문 수용
2. 시맨틱 모델 정의
3. 진단 추가
4. 참조 동작 구현
5. 픽스처 테스트 추가
6. 최적화는 나중에 추가

### 3.3 구현 규칙

다음을 포함하지 않으면 어떤 단계도 완료로 간주하지 않는다:

- 소스 픽스처
- 예상 진단 또는 출력 스냅샷
- 검사 또는 실행을 위한 CLI 표면
- 다음 단계가 지연될 경우를 위한 명확한 롤백 경계

## 4. MVP 동결

### 4.1 MVP 포함 항목

- 다중 파일 모듈 로딩
- 렉서, 파서, AST
- 파일/라인/컬럼 스팬이 포함된 진단
- 이름 해결 및 렉시컬 스코프
- 문서화된 핵심 타입에 대한 타입 추론
- `@html`, `@server`, `@design` 도메인 검증
- 타입화된 객체 리터럴 및 패턴 매칭
- 참조 런타임:
  - html 트리 평가
  - 시그널
  - 라우트
  - 요청 접근자
  - 응답
  - `@serve`
  - `@env`
- CLI 명령어:
  - `orv version`
  - `orv check`
  - `orv dump ast`
  - `orv dump hir`
  - `orv run`
  - `orv build`
- `.fetch()`를 위한 JSON 기반 풀스택 RPC v1
- 정적 html 빌드 및 개발 서버 빌드 타겟

### 4.2 MVP 제외 항목

- 바이너리 RPC 전송
- 어댑티브 런타임
- 공격적인 dead code elimination
- 프로덕션 수준 청크 분할
- WASM 백엔드
- 네이티브 최적화 백엔드
- 범용 외부 IDE 동등성
- 포매터
- 증분 컴파일 캐시

### 4.3 MVP 성공 정의

문서에 있는 작은 다중 파일 orv 앱이 다음을 수행할 수 있어야 한다:

1. 파싱
2. 타입 체크
3. 유효하지 않은 도메인 사용 거부
4. 참조 런타임을 통한 실행
5. 기본 라우트 서빙
6. 기본 페이지 렌더링
7. `.fetch()`를 통한 라우트 호출

## 5. 목표 크레이트 아키텍처

| 크레이트 | 목적 | 초기 단계 |
|------|---------|---------------|
| `orv-cli` | 커맨드라인 오케스트레이션 | 기존 |
| `orv-core` | 공개 파사드 크레이트 | 기존 |
| `orv-macros` | proc-macro 파사드, 향후 선택적 통합 | 기존 |
| `orv-span` | 파일 ID, 스팬, 소스 텍스트 인덱싱 | Phase 0 |
| `orv-diagnostics` | 오류, 경고, 리포트, 출력 | Phase 0 |
| `orv-syntax` | 렉서, 파서, AST | Phase 1-2 |
| `orv-hir` | 하위 시맨틱 IR | Phase 3-5 |
| `orv-analyzer` | 스코프, 타입, 도메인 검증 | Phase 3-5 |
| `orv-project` | 모듈 그래프 및 ProjectGraph | Phase 5-6 |
| `orv-runtime` | 참조 런타임 및 인터프리터 | Phase 7-9 |
| `orv-compiler` | 빌드 파이프라인, emit, 백엔드 | Phase 8-12 |
| `orv-ide` | 에디터 대응 쿼리, 자동완성, 호버, 스테퍼 | MVP 이후 기반 |

## 6. 마일스톤 계획

### Phase 0. 기반 및 픽스처

**목표**

나중에 아키텍처를 다시 그리지 않고 언어를 구축하는 데 필요한 지원 시스템을 생성한다.

**산출물**

- `orv-span`
- `orv-diagnostics`
- 픽스처 하네스
- 스냅샷 테스트 규칙
- 소스 파일 로더 추상화

**티켓**

1. `FileId`, `Span`, `Spanned<T>`, 라인 인덱스 조회를 포함한 `orv-span` 추가.
2. `Severity`, `Diagnostic`, 라벨, 노트, 도움말 텍스트를 포함한 `orv-diagnostics` 추가.
3. 픽스처 디렉토리 추가:
   - `fixtures/lexer`
   - `fixtures/parser`
   - `fixtures/analyzer`
   - `fixtures/runtime`
4. 골든 출력 비교를 위한 테스트 헬퍼 추가.
5. 픽스처 기반 개발을 위한 `orv-cli` 숨김 디버그 명령 추가.

**주요 시나리오**

- 잘못된 형식의 소스 파일이 정확한 위치를 보고한다.
- 프로젝트의 여러 파일을 `FileId`로 로드하고 식별할 수 있다.
- 진단이 주요 및 보조 스팬을 가리킬 수 있다.

**엣지 케이스**

- 주석만 있는 파일
- 빈 파일
- 후행 개행이 없는 파일
- 매우 긴 라인
- 문자열 리터럴의 유니코드
- 양쪽 파일을 가리키는 노트가 있는 크로스 파일 import 오류

**종료 기준**

- 진단이 테스트에서 결정적으로 렌더링된다
- 스팬 수학이 단위 테스트로 검증된다
- 렉서 구현 시작 전에 픽스처 러너가 존재한다

### Phase 1. 렉서

**목표**

소스 텍스트를 라인 지향 파싱에 충분한 구조를 보존하는 안정적인 토큰 스트림으로 변환한다.

**산출물**

- 토큰 열거형
- 스팬 방출이 포함된 렉서
- 줄바꿈 및 들여쓰기 처리
- 주석 제거 및 트리비아 정책

**티켓**

1. 식별자, 키워드, 구두점, 리터럴, 연산자, 보간 마커에 대한 토큰 타입 정의.
2. 개행 처리 전략 결정:
   - `Newline` 방출
   - 들여쓰기 카운트 보존
   - 인라인 `%` 프로퍼티에 대한 연속 라인 허용
3. 보간 세그먼트가 포함된 문자열 리터럴 지원 구현.
4. 라우트 및 힌트 구문을 위한 경로/메서드/속성 토큰화 구현.
5. 닫히지 않은 구조와 유효하지 않은 문자에 대한 진단 방출.

**주요 시나리오**

- `@button "Click" %onClick={count += 1}`
- 노드 선언 후 연속 프로퍼티 라인
- `/users/:id` 같은 라우트 경로
- `@hint protocol=json`
- `"Hello {name}"`

**엣지 케이스**

- 종료되지 않은 문자열
- 보간 내부의 매칭되지 않은 `{`
- 연결 가능한 부모가 없는 최상위 `%`
- 연속 라인에서 탭과 스페이스 혼용
- 떠도는 `@` 토큰
- `@token \d+`의 정규식 유사 토큰

**종료 기준**

- 모든 파서 픽스처가 안정적인 토큰 스냅샷에서 시작한다
- 렉서가 퍼징된 작은 입력에서 패닉하지 않는다

### Phase 2. 파서 및 AST

**목표**

구문 레퍼런스에 문서화된 언어 표면을 복구 가능한 AST로 파싱한다.

**산출물**

- AST 정의
- 복구가 포함된 파서
- AST 덤프 명령

**티켓**

1. 모듈, 항목, 문, 표현식, 타입, 패턴, 노드, 프로퍼티 AST 정의.
2. 최상위 항목 파싱:
   - `import`
   - `type`
   - `struct`
   - `enum`
   - `function`
   - `define`
   - 최상위 `let`/`const`
3. 노드 구문 파싱:
   - 위치 토큰
   - 인라인 프로퍼티
   - 연속 프로퍼티
   - 본문 블록
4. 첫 번째 문 규칙을 사용한 객체 리터럴 vs 코드 블록 파싱.
5. `when`, `try/catch`, `async`, 튜플 await, 제네릭, 라우트 힌트 파싱.
6. 중괄호, 소괄호, 노드 블록 주변에 복구 추가.

**주요 시나리오**

- 문서의 모든 `orv` 펜스드 예제가 파싱된다
- 구문 덤프가 스냅샷 테스트에 충분히 안정적이다
- 부분적으로 깨진 파일이 향후 에디터 도구를 위해 사용 가능한 AST를 생성한다

**엣지 케이스**

- 쉼표가 있거나 없는 다중 라인 객체 리터럴
- `@respond 200 %header={...} { ... }`
- `pub define X() -> @html @hint render=ssr { ... }`
- 중첩된 `define`
- 패턴 위치의 `Point { x, y }`
- 표현식 뒤의 모호한 `{}`

**종료 기준**

- `orv dump ast`가 모든 정상 경로 픽스처에서 동작한다
- 구문 오류가 최소 하나의 하위 오류를 방출할 수 있을 만큼 복구 가능하다

### Phase 3. 이름 해결 및 모듈 그래프

**목표**

진정한 렉시컬 스코프 규칙 하에서 식별자와 import를 해결한다.

**산출물**

- 모듈 그래프 로더
- import 해결기
- 심볼 테이블
- 렉시컬 스코프 맵

**티켓**

1. 파일 레이아웃에서 모듈 경로 해결 구현.
2. 내보내기와 비공개 선언 해결.
3. 로컬 바인딩, 패턴, 함수 파라미터, `define` 파라미터 해결.
4. 라우트 참조 및 동일 스코프 `.fetch()` 대상 해결.
5. 중복 심볼 및 미해결 심볼 오류 감지.
6. 직접 import 순환 감지.

**주요 시나리오**

- import된 `pub define`
- `if`/`for` 내부의 로컬 심볼 쉐도잉
- `@children` 및 내부 `define` 이름
- 동일 렉시컬 스코프에서 형제 코드가 사용하는 라우트 참조

**엣지 케이스**

- import 별칭 충돌
- 와일드카드 import 충돌
- 순환 재내보내기
- 선언 전 사용 정책
- 외부 이름을 쉐도잉하는 패턴 바인딩

**종료 기준**

- 분석기 픽스처의 모든 이름이 바인딩되었거나 명확하게 진단된다
- 라우트 스코프 동작이 임의의 런타임 매직이 아닌 문서와 일치한다

### Phase 4. 타입 시스템

**목표**

진단을 의미 있게 만들고 런타임 하위화를 안전하게 만드는 데 필요한 핵심 타입 규칙을 구현한다.

**산출물**

- 타입 표현
- 추론 엔진
- 제약 조건 해결기
- 타입 진단

**티켓**

1. 프리미티브, nullable, 튜플, 함수, struct, enum, 제네릭, `void` 타입 추가.
2. 리터럴 및 표현식에 대한 로컬 추론 구현.
3. 타입화된 객체 리터럴 검증 구현.
4. `when`에 대한 패턴 타이핑 구현.
5. 함수, 클로저, `.fetch()`에 대한 호출 가능 타이핑 구현.
6. `@env` 컨텍스트 타이핑 구현.

**주요 시나리오**

- `let port: i32 = @env PORT`
- `let sig remaining = todos.len()`
- `when result { Status.Ok(code) -> ... }`
- `let user: User = { name: "x", age: 1 }`

**엣지 케이스**

- 컨텍스트 없는 빈 `[]`
- 예상 타입이 없는 객체 리터럴로 사용된 빈 `{}`
- `if x != void` 이후의 nullable 축소
- `when`의 혼합 타입 분기
- 연관 enum 페이로드 불일치
- 객체 필드 누락 또는 초과

**종료 기준**

- 문서의 기초 예제가 타입 체크되거나 의도적인 진단으로 실패한다
- 암시적 `any`가 없다

### Phase 5. 도메인 검증

**목표**

도메인 오용이 컴파일 타임 오류라는 언어의 핵심 약속을 시행한다.

**산출물**

- 도메인 컨텍스트 모델
- 노드/프로퍼티 유효성 매트릭스
- 도메인 인식 진단

**티켓**

1. 도메인 컨텍스트 모델링:
   - neutral
   - html
   - server
   - design
2. 각 컨텍스트에 대한 유효한 노드와 프로퍼티 정의.
3. 전환 및 크로스 도메인 참조 검증.
4. 올바른 도메인 가정 하에서 이벤트 핸들러와 콜백 본문 검증.
5. html/파일/정적 대상에 대한 `@serve` 규칙 검증.

**주요 시나리오**

- 서버 노드만 포함하는 `@server`
- html 안전 노드만 포함하는 `@html`
- `page`가 html인 `@serve page`
- UI에서의 디자인 토큰 사용

**엣지 케이스**

- 도메인별 노드를 반환하는 `define`
- import된 정의 뒤에 숨겨진 도메인 불일치
- 선언된 것과 다른 도메인에 삽입된 `@children`
- 잘못된 콜백 스코프 내부의 `@route`

**종료 기준**

- 유효하지 않은 도메인 픽스처가 결정적으로 거부된다
- 문서 예제에 대한 거짓 양성이 허용 가능한 임계값 이하이다

### 7.1 테스트 레이어

| 레이어 | 목적 | 위치 |
|------|---------|----------|

| 런타임 시나리오 | 실행 가능한 언어 동작 | `fixtures/runtime` |
| 빌드 아티팩트 | 방출된 파일 및 매니페스트 | `fixtures/build` |

- 런타임 동작은 최적화 없이도 재현 가능해야 한다

## 8. 기능 시나리오 매트릭스

### 시나리오 A. 정적 HTML 페이지

**소스**

- `@html`, `@head`, `@body`가 포함된 파일 하나

**예상 동작**

- 깔끔하게 파싱된다
- html 도메인으로 검증된다
- 정적 html로 빌드된다
- 서버 런타임이 필요하지 않다

**실패 모드**

- html 컨텍스트에서 유효하지 않은 노드
- 누락된 필수 보간 심볼
- 정적 분류기가 런타임 의존성을 잘못 유지

### 시나리오 B. 시그널 카운터

**소스**

- `let sig count = 0`
- 텍스트 보간
- 버튼 이벤트가 count를 증가시킴

**예상 동작**

- 시그널 저장소가 생성된다
- 버튼 이벤트 클로저가 시그널을 변경한다
- 영향받는 텍스트 노드만 업데이트된다

**실패 모드**

- 이벤트 클로저가 잘못된 바인딩을 캡처
- 시그널 그래프가 자기 루프를 생성
- 대상 업데이트 대신 전체 리렌더

### 시나리오 C. 기본 서버

**소스**

- `@server`
- `@listen`
- `@route GET /`
- `@respond 200 { ... }`

**예상 동작**

- 라우트 디스패치가 동작한다
- 추론된 메타데이터와 함께 JSON 응답이 방출된다

**실패 모드**

- 라우트 충돌
- 유효하지 않은 응답 페이로드 형태
- 라우트 스코프 외부에서 사용된 요청 접근자

### 시나리오 D. 풀스택 RPC

**소스**

- `let getUsers = @route GET /api/users { ... }`
- 동일 스코프 html에서 `await getUsers.fetch()` 호출

**예상 동작**

- 분석기가 라우트 심볼을 해결한다
- `.fetch()` 인수가 검증된다
- 런타임이 라우트를 호출하고 타입화된 값을 반환한다

**실패 모드**

- 렉시컬 스코프 외부에서 사용된 라우트
- 잘못된 `param` 키
- 라우트 참조가 아닌 심볼에서의 `.fetch()`

### 시나리오 E. `@env` 추론

**소스**

- `let port: i32 = @env PORT`
- `let secret = @env SECRET`

**예상 동작**

- `port`가 컨텍스트 강제 변환을 받는다
- `secret`은 문자열로 유지된다

**실패 모드**

- 누락된 env 변수
- 잘못된 형식의 정수 문자열
- 예상 타입이 없는 암시적 강제 변환

### 시나리오 F. `@hint` 왕복

**소스**

- `@hint`가 포함된 라우트 및 페이지

**예상 동작**

- 파서가 힌트를 수용한다
- 분석기가 허용된 대상을 검증한다
- 고급 옵티마이저가 존재하지 않더라도 빌드 메타데이터가 힌트를 보존한다

**실패 모드**

- 잘못된 대상에 부착된 힌트
- 지원되지 않는 힌트 값이 조용히 무시됨

## 9. 사전 부검

### 실패 시나리오 1. 문법 모호성이 프론트엔드를 태운다

**위험**

객체 리터럴, 코드 블록, 연속 라인, 노드 본문이 미묘한 방식으로 충돌한다.

**완화**

- 파서 작업 전에 픽스처 코퍼스를 동결
- 모호한 구조를 명시적으로 테스트
- 문서화되지 않은 숨겨진 파서 휴리스틱 회피

### 실패 시나리오 2. 런타임 시맨틱이 향후 백엔드와 드리프트한다

**위험**

참조 런타임이 빌드 출력과 다르게 동작하여 최적화가 안전하지 않게 된다.

**완화**

- 런타임을 시맨틱 오라클로 취급
- 백엔드 동작을 런타임 픽스처와 비교
- 런타임 동작이 안정되기 전에 기능을 최적화하지 않음

### 실패 시나리오 3. RPC 기능 하에서 스코프 규칙이 퇴행한다

**위험**

"예제를 동작시키기 위해" 라우트 참조가 실수로 전역 또는 동적이 된다.

**완화**

- 라우트 참조에 대한 스코프 해결 테스트
- `.fetch()`가 렉시컬 바인딩만 사용한다는 명시적 분석기 규칙
- 문서화되지 않은 스코프 리프팅 거부

## 11. 결정 게이트

다음 단계를 시작하기 전에 명시적으로 확인한다:

### 게이트 A. 이름 해결 전

- import 해결 동작이 명확하다

- 타입 진단이 신뢰할 수 있다

- 참조 런타임 시맨틱이 안정적이다
- HIR과 ProjectGraph가 발밑에서 움직이지 않는다

### 게이트 E. 고급 최적화 전

- 핵심 앱 시나리오가 최적화 없이 이미 올바르게 실행된다
