# 왜 orv인가

[← 사용자 문서로 돌아가기](./README.ko.md)

---

## 문제

현대 소프트웨어 스택은 각각 독립적으로 최적화된 도메인별 시스템들을 조합하여 구성됩니다:

- 데이터베이스는 저장과 쿼리에 최적화됩니다
- 전송 계층은 범용 상호운용성에 최적화됩니다
- UI 프레임워크는 자체 런타임 내의 렌더링 패턴에 최적화됩니다
- 서버 프레임워크는 자체 추상화 모델 내의 요청 처리에 최적화됩니다

이러한 분리는 이식성을 가져왔지만, 동시에 프로젝트가 훨씬 좁고 예측 가능한 형태를 가지고 있음에도 개발자가 호환성과 범용 도구에 대한 비용을 자주 지불해야 한다는 것을 의미합니다.

## 관찰

대부분의 프로젝트는 최대한의 범용성을 필요로 하지 않습니다. 랜딩 페이지, 대시보드, 내부 도구, 게임 유사 웹 앱은 각각 다른 제약 조건을 가지고 있지만, 많은 스택이 여전히 동일한 광범위한 프로토콜, 번들링 가정, 프레임워크 경계에 의존합니다.

팀이 도메인별 도구를 채택하더라도, 여전히 수동으로 조사하고 조합해야 합니다. 프로젝트 목표는 스택 외부에 남아 있습니다.

## 핵심 아이디어

orv는 다른 전제에서 출발합니다:

> 프레임워크와 컴파일러가 도메인만이 아니라 프로젝트 자체에 최적화된다면 어떨까?

컴파일러가 전체 프로젝트의 구조를 볼 수 있다면, 경계를 넘어 결정을 내릴 수 있습니다:

- UI와 서버를 함께 최적화할 수 있습니다
- 데이터 흐름이 렌더 전략을 결정할 수 있습니다
- 라우트 사용 패턴이 전송 형태를 결정할 수 있습니다
- 실제 프로젝트 의도가 번들링과 데드 코드 제거를 주도할 수 있습니다

결과물은 범용 상호운용성 기준이 아닌, 만들고 있는 애플리케이션에 맞춰 형성된 빌드 산출물입니다.

## 트레이드오프

이 접근 방식은 더 긴밀한 통합과 더 강력한 최적화를 위해 일부 범용 호환성을 의도적으로 포기합니다. 이 교환은 생태계가 충분한 생산성과 성능 가치를 제공하여 정당화할 수 있을 때 가치가 있습니다.

orv는 이 트레이드오프를 중심으로 설계되었습니다.

## 언어 방향

언어가 선언적이고 라인 지향적인 것은 의도적입니다:

- 각 줄은 하나의 명확한 의도를 전달해야 합니다
- 도메인 관계는 읽기 쉽게 유지되어야 합니다
- 구문은 사람과 구조적 분석을 수행하는 AI 시스템 모두에게 이해 가능해야 합니다

이것이 언어 레퍼런스가 명시적 구조, 렉시컬 스코프, 도메인 인식 유효성 검사를 강조하는 이유입니다.

## 내장 에디터 방향

orv는 자체 에디터 경험을 함께 제공할 예정입니다.

그 이유는 미적인 것이 아니라 구조적인 것입니다:

- 언어가 이미 `@node`를 통해 도메인을 구분합니다
- 노드 구성이 이미 `%property`를 통해 흐릅니다
- 따라서 컴파일러는 커서 위치에서 어떤 노드와 프로퍼티가 유효한지 알 수 있습니다

이를 통해 다음과 같은 에디터를 구축할 수 있습니다:

- `@`를 입력하면 도메인 인식 노드 선택지가 열립니다
- `%`를 입력하면 노드 인식 프로퍼티 선택지가 열립니다
- 유틸리티 크기나 숫자 리터럴 같은 값의 단계 조정이 텍스트적이 아닌 의미론적으로 이루어집니다

이는 에디터 지원을 얇은 외부 IDE 플러그인 레이어로 취급하는 것보다 orv에 더 적합한 방식입니다.
