## Philosophy

orv는 여섯 가지 원칙 위에 구축되었습니다:

- **하나의 구문으로 모든 도메인을.** UI, 서버, 디자인 토큰, 범용 로직이 통합된 문법을 공유합니다. `@node` / `%property` 구조는 버튼부터 HTTP 서버까지 확장됩니다.
- **재사용 가능한 도메인 추상화: `define`.** `class`, `new`, `this`, 상속이 없습니다. `define`은 컴포넌트, 라우트 그룹, 전체 도메인 패밀리까지 포함하는 재사용 가능한 `@domain` 표면 문법을 설명합니다.
- **마법 없는 간결함.** 모든 축약형에는 예측 가능한 확장이 있습니다. `$0`은 항상 첫 번째 콜백 매개변수입니다. `sig`는 항상 반응형 시그널입니다. 숨겨진 변환은 없습니다.
- **컴파일 타임 안전성, 런타임 속도.** 타입은 Rust처럼 추론되고, 컴파일 타임에 검사되며, WASM(웹) 또는 네이티브 바이너리로 컴파일됩니다. 도메인 컨텍스트는 컴파일 타임에 검증되므로 `@server` 내부에 `@div`를 넣을 수 없습니다.
- **프로젝트별 최적화.** 컴파일러는 전체 프로젝트 — UI, 서버, 디자인, 그리고 그들의 관계 — 를 분석하여 프로젝트의 실제 목적에 최적화된 번들을 생성합니다. 범용 프로토콜과 포맷에 의존하는 대신, 프로젝트가 사용하는 특정 도메인과 통신 패턴에 맞춤화된 출력을 생성합니다.
- **선언적이고 공백-구조화된 줄 중심 문법.** 각 줄은 하나의 명확한 의도를 가집니다. 공백은 node head를 subtoken/property/data로 분할하고, 줄바꿈은 statement를 분할합니다. 사람과 AI가 줄 단위로 읽어도 의미가 모호하지 않도록 설계되었습니다.
- **에디터 네이티브 구조.** 이 언어는 의도적으로 orv 전용 내장 에디터에 맞게 설계되었습니다. `@`는 현재 도메인에서 유효한 노드 선택지를 열어야 하고, `%`는 현재 노드에서 유효한 속성을 열어야 하며, 시맨틱 값은 일반 텍스트 편집으로 되돌아가지 않고 단계별로 조작할 수 있어야 합니다.
